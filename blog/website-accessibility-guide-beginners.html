Website Accessibility Guide for Beginners: WCAG 2.2, ARIA, and Inclusive Design in 2026

Share on XShare on FacebookShare on LinkedInShare on Reddit

Over 1 billion people worldwide live with some form of disability, according to the World Health Organization. In the digital context, this includes visual impairments (blindness, low vision, color blindness), hearing impairments, motor disabilities (inability to use a mouse), cognitive disabilities (dyslexia, ADHD), and temporary disabilities (broken arm, situational impairments). Web accessibility ensures all of these users can perceive, understand, navigate, and interact with your website.

Beyond the ethical imperative, accessibility is increasingly a legal requirement. In the United States, ADA (Americans with Disabilities Act) lawsuits against websites have increased from 2,314 in 2018 to over 4,000 annually by 2025. The European Accessibility Act (EAA) mandates web accessibility compliance by June 2025 for businesses operating in the EU. This guide covers everything beginners need to know about making websites accessible in 2026.

Understanding WCAG 2.2

WCAG (Web Content Accessibility Guidelines) is the international standard for web accessibility, developed by the W3C's Web Accessibility Initiative. WCAG 2.2, published in October 2023, is the current version. It is organized around four principles, known by the acronym POUR:

PrincipleMeaningExamples
PerceivableUsers can perceive all contentAlt text for images, captions for video, sufficient color contrast
OperableUsers can navigate and interactKeyboard accessibility, sufficient time limits, no seizure-inducing content
UnderstandableContent and UI are understandableClear language, predictable navigation, error identification
RobustContent works with assistive technologiesValid HTML, ARIA labels, compatible with screen readers

Compliance Levels

  • Level A (Minimum): 30 success criteria addressing the most critical barriers. Without Level A compliance, some users will be completely unable to access content.
  • Level AA (Recommended): 24 additional criteria. This is the standard targeted by most legal requirements, including ADA, Section 508, and EAA. Level AA addresses barriers that create significant difficulty for users with disabilities.
  • Level AAA (Highest): 28 additional criteria. Full AAA compliance is not typically required and may not be achievable for all content types. It represents the highest level of accessibility.

New in WCAG 2.2

WCAG 2.2 added 9 new success criteria, primarily addressing mobile accessibility and cognitive disabilities. Key additions include: Focus Not Obscured (ensuring focused elements are not hidden behind sticky headers or overlays), Dragging Movements (providing alternatives to drag interactions), Target Size (minimum 24x24 CSS pixels for interactive targets), and Consistent Help (placing help mechanisms in consistent locations).

Semantic HTML: The Foundation

Semantic HTML is the single most impactful accessibility practice. Using the correct HTML elements communicates the structure and purpose of content to assistive technologies without any additional ARIA attributes or JavaScript.

Key Semantic Elements

  • <header>, <nav>, <main>, <footer>: Landmark elements that allow screen reader users to jump directly to major page sections. A screen reader user can press a shortcut to list all landmarks and navigate to the one they need.
  • <h1> through <h6>: Headings create an outline that screen reader users navigate like a table of contents. Maintain proper hierarchy (never skip from h1 to h3). Use only one h1 per page.
  • <button> vs <div>: Native button elements are keyboard-focusable, activatable with Enter and Space, and announced as "button" by screen readers. A div with an onclick handler has none of these features without extensive ARIA and JavaScript.
  • <a> for links, <button> for actions: Links navigate to a new URL or page section. Buttons perform an action on the current page. Using the correct element ensures screen readers announce the correct function.
  • <ul>, <ol>, <dl>: List elements tell screen readers how many items are in a group and where the user is within the list ("item 3 of 7").
  • <table> with <th>: Data tables with proper header cells allow screen readers to announce column and row headers as users navigate cells.
The Div Soup Problem

Many modern websites are built almost entirely with div and span elements styled to look like buttons, links, headings, and lists. While these look correct visually, they provide zero semantic information to assistive technologies. A screen reader encounters them as generic text with no indication of purpose or interactivity. Always use the native HTML element that matches the content's purpose.

Keyboard Navigation

Approximately 2% of web users rely on keyboard-only navigation due to motor disabilities, screen reader usage, or personal preference. Keyboard accessibility is a Level A requirement and is relatively straightforward to implement correctly.

Essential Keyboard Interactions

  • Tab: Moves focus to the next interactive element (links, buttons, form fields).
  • Shift+Tab: Moves focus to the previous interactive element.
  • Enter: Activates links and buttons.
  • Space: Activates buttons, toggles checkboxes, opens dropdowns.
  • Escape: Closes modals, menus, and overlays.
  • Arrow keys: Navigate within components (tabs, menus, radio groups).

Focus Management

Every interactive element must have a visible focus indicator. The default browser outline (blue ring in Chrome, dotted border in Firefox) is functional but can be styled with CSS. Never use outline: none without providing an alternative focus style. WCAG 2.2 requires focus indicators to have at least a 3:1 contrast ratio against adjacent colors.

Focus Trapping for Modals

When a modal dialog opens, keyboard focus must be trapped within it -- Tab should cycle through the modal's interactive elements without escaping to the background content. When the modal closes, focus must return to the element that opened it. This prevents keyboard users from getting lost behind an overlay they cannot see or interact with.

Images and Alt Text

Every img element requires an alt attribute. The content of that attribute depends on the image's purpose:

Informative Images

Images that convey information need descriptive alt text. Describe what the image shows and why it matters in context. Keep it concise (typically under 125 characters). Example: alt="Bar chart showing website traffic increasing from 10,000 to 45,000 monthly visitors between January and June 2026"

Decorative Images

Images that are purely visual decoration (background patterns, spacers, decorative borders) should have empty alt text: alt="". This tells screen readers to skip the image entirely. An absent alt attribute causes screen readers to announce the filename, which is worse than no description.

Functional Images

Images used as links or buttons should have alt text describing the function, not the image. A logo linking to the homepage: alt="AliensWebsite home", not alt="AliensWebsite logo". A magnifying glass icon for search: alt="Search", not alt="magnifying glass icon".

Color and Contrast

Approximately 8% of men and 0.5% of women have some form of color vision deficiency. Relying on color alone to convey information excludes these users.

Contrast Requirements

Content TypeLevel AALevel AAA
Normal text (<18px)4.5:17:1
Large text (18px+ / 14px+ bold)3:14.5:1
UI components & graphics3:13:1

Color Independence

Never use color as the only way to convey information. If error messages are shown in red, also add an error icon or text label. If links are distinguished from body text only by color, also add an underline. If chart data uses different colors for categories, also use different patterns or labels.

ARIA Labels and Roles

ARIA (Accessible Rich Internet Applications) provides attributes that enhance accessibility when native HTML semantics are insufficient. However, ARIA is frequently misused, and incorrect ARIA is worse than no ARIA at all.

The Five Rules of ARIA

  1. Do not use ARIA if a native HTML element provides the same semantics. Use <button> instead of <div role="button">.
  2. Do not change native semantics. Do not add role="heading" to a button element.
  3. All interactive ARIA controls must be keyboard operable. If you add role="button" to a div, you must also handle Enter, Space, and focus.
  4. Do not use aria-hidden="true" on focusable elements. This creates an element that screen readers cannot see but keyboard users can reach.
  5. All interactive elements must have an accessible name. Via visible text, alt attribute, aria-label, or aria-labelledby.

Common ARIA Attributes

  • aria-label: Provides an accessible name when no visible text exists. Example: <button aria-label="Close menu">X</button>
  • aria-labelledby: Points to another element whose text serves as the accessible name. Useful for headings that label sections.
  • aria-describedby: Points to an element providing additional description. Useful for form field help text.
  • aria-hidden="true": Hides an element from screen readers. Use for decorative elements that should not be announced.
  • aria-expanded: Indicates whether a collapsible section is open or closed.
  • aria-live: Marks a region whose content updates dynamically. Screen readers announce changes to live regions. Use "polite" for non-urgent updates and "assertive" for critical alerts.

Accessibility Testing Tools

Automated Tools

  • WAVE (wave.webaim.org): The most visual accessibility checker. Shows errors, alerts, and features as icons overlaid on your page. Free browser extensions available for Chrome and Firefox.
  • axe DevTools (browser extension): The most technically precise automated checker. Developed by Deque, it catches WCAG violations with zero false positives by design. Used by Microsoft, Google, and the US government.
  • Lighthouse (Chrome DevTools): Built into Chrome browser. Provides an accessibility score from 0-100 based on a subset of axe-core rules. Good for quick checks but less comprehensive than dedicated tools.
  • Pa11y: Command-line accessibility testing tool that can be integrated into CI/CD pipelines. Catches regressions before they reach production.

Manual Testing

  • Keyboard-only navigation: Unplug your mouse and navigate your entire site using only Tab, Enter, Escape, and Arrow keys. Can you reach every interactive element? Is focus visible? Can you escape modals?
  • Screen reader testing: VoiceOver (built into macOS, Cmd+F5 to activate), NVDA (free, Windows), JAWS (paid, Windows). Navigate your site and verify that all content is announced logically.
  • Zoom testing: Zoom your browser to 200% and verify the layout remains usable and no content is cut off or overlapping.
  • Color blindness simulation: Chrome DevTools has built-in color blindness simulation (Rendering tab > Emulate vision deficiencies).
Start Small: The Top 5 Accessibility Fixes

1. Add alt text to all images. 2. Use semantic HTML (headings, buttons, landmarks). 3. Ensure all interactive elements are keyboard accessible. 4. Check color contrast ratios (4.5:1 minimum for text). 5. Add visible focus indicators. These five changes address the majority of accessibility barriers on most websites and can typically be implemented in a single day.

Frequently Asked Questions

What is WCAG 2.2 and do I need to comply?
WCAG 2.2 is the international standard for web accessibility published by the W3C. It has three levels: A, AA, and AAA. Most legal requirements target Level AA. ADA lawsuits against websites exceed 4,000 annually in the US. The European Accessibility Act mandates compliance for EU businesses. Compliance protects you legally and ensures access for all users.
How do I test my website for accessibility?
Combine automated and manual testing. Automated: WAVE provides visual error overlays; axe DevTools catches WCAG violations; Lighthouse includes accessibility audits. Manual: navigate with keyboard only, test with a screen reader (VoiceOver on Mac, NVDA on Windows), check color contrast. Automated tools catch only 30-40% of issues -- manual testing is essential.
What is the minimum color contrast ratio?
WCAG Level AA requires 4.5:1 for normal text and 3:1 for large text (18px+ or 14px+ bold). Level AAA requires 7:1 and 4.5:1 respectively. UI components need 3:1 contrast. Use the WebAIM Contrast Checker to verify.
Do I need alt text on every image?
Every img element needs an alt attribute, but not all need descriptive text. Informative images need descriptions. Decorative images use empty alt (alt=""). Functional images (links/buttons) describe the function, not the appearance. Complex images need detailed text alternatives.
What are ARIA labels and when should I use them?
ARIA attributes provide accessibility information when native HTML is insufficient. Common ones: aria-label (accessible name), aria-hidden (hide from screen readers), aria-expanded (collapsible state). The first rule: do not use ARIA if a native HTML element works. Use button instead of div role="button".
